57 class BaseCommand(BaseCommandOriginal, OutputBase):
58 """
59 The base class from which all management commands ultimately derive.
60  
61 Use this class if you want access to all of the mechanisms which
62 parse the command-line arguments and work out what code to call in
63 response; if you don't need to change any of that behavior,
64 consider using one of the subclasses defined in this file.
65  
66 If you are interested in overriding/customizing various aspects of
67 the command-parsing and -execution behavior, the normal flow works
68 as follows:
69  
70 1. ``django-admin`` or ``manage.py`` loads the command class
71 and calls its ``run_from_argv()`` method.
72  
73 2. The ``run_from_argv()`` method calls ``create_parser()`` to get
74 an ``ArgumentParser`` for the arguments, parses them, performs
75 any environment changes requested by options like
76 ``pythonpath``, and then calls the ``execute()`` method,
77 passing the parsed arguments.
78  
79 3. The ``execute()`` method attempts to carry out the command by
80 calling the ``handle()`` method with the parsed arguments; any
81 output produced by ``handle()`` will be printed to standard
82 output and, if the command is intended to produce a block of
83 SQL statements, will be wrapped in ``BEGIN`` and ``COMMIT``.
84  
85 4. If ``handle()`` or ``execute()`` raised any exception (e.g.
86 ``CommandError``), ``run_from_argv()`` will instead print an error
87 message to ``stderr``.
88  
89 Thus, the ``handle()`` method is typically the starting point for
90 subclasses; many built-in commands and command types either place
91 all of their logic in ``handle()``, or perform some additional
92 parsing work in ``handle()`` and then delegate from it to more
93 specialized methods as needed.
94  
95 Several attributes affect behavior at various steps along the way:
96  
97 ``help``
98 A short description of the command, which will be printed in
99 help messages.
100  
101 ``output_transaction``
102 A boolean indicating whether the command outputs SQL
103 statements; if ``True``, the output will automatically be
104 wrapped with ``BEGIN;`` and ``COMMIT;``. Default value is
105 ``False``.
106  
107 ``requires_migrations_checks``
108 A boolean; if ``True``, the command prints a warning if the set of
109 migrations on disk don't match the migrations in the database.
110  
111 ``requires_system_checks``
112 A list or tuple of tags, e.g. [Tags.staticfiles, Tags.models]. System
113 checks registered in the chosen tags will be checked for errors prior
114 to executing the command. The value '__all__' can be used to specify
115 that all system checks should be performed. Default value is '__all__'.
116  
117 To validate an individual application's models
118 rather than all applications' models, call
119 ``self.check(app_configs)`` from ``handle()``, where ``app_configs``
120 is the list of application's configuration provided by the
121 app registry.
122  
123 ``stealth_options``
124 A tuple of any options the command uses which aren't defined by the
125 argument parser.
126 """
